# 架构说明

## 数据流

1. 主进程包装 QQ `webContents.send`，只读取本地 Electron IPC 中的消息列表。
2. 支持的消息进入 10,000 条先进先出内存缓存；图片和表情候选先保留真实元素与路径，撤回时再判断本地文件是否有效，避免 QQ 文件晚于消息事件落盘时提前丢弃候选。
3. 收到撤回灰条时，以消息 ID 查找原消息；保留撤回包的序列、客户端序列和发送状态，并恢复原消息的 `msgAttrs`、`msgMeta`、`generalFlags` 特殊类型后发往渲染层。
4. 对仅靠页面快照恢复的图片/表情，渲染层只上报受限 `appimg:` URL；其他来源只允许 Canvas PNG。主进程按魔数识别并以 SHA-256 保存到 `media/`，再原子更新对应 JSON 引用。
5. QQ 重启后，撤回灰条再次出现时，从本地记录恢复消息，并按当前记录根校验媒体相对路径、大小和 SHA-256 后重建本地路径。

图片与图片表情恢复优先复用 `picElement` 已携带的有效本地路径；商城表情优先复用 `marketFaceElement` 的本地路径。仅靠页面快照恢复时标记为 `memoryOnly`，随后优先读取严格位于 QQ `Pic/Emoji` 下的 `appimg:` 原件并保留动画；原件不可读时降级为静态 PNG。两条路径均失败时继续保留当前会话行为。插件不调用网络接口，也不补下载缺失文件。

## 存储

当前记录根包含 `records/` 和 `media/`：媒体单项上限 20 MiB，按真实 GIF/PNG/JPEG/WebP 魔数确定扩展名并以 SHA-256 去重。设置页面选择新目录时先复制仍被记录引用的媒体，再切换记录和媒体根；孤儿媒体不迁移，失败时旧根继续生效，旧目录不会自动删除。

管理窗口通过独立的 `qqLocalRecallManager` Preload 桥接获得固定接口，不能传入或读取任意文件路径。删除会话时同时更新记录、内存候选和媒体引用；共享内容仍被其他记录引用时保留，最后引用删除后清理媒体文件。

## 渲染层

渲染层接收已恢复消息 ID、内容类型、操作者角色/名称、原发送者名称、`memoryOnly` 和已删除消息 ID。`operatorRole=0/1/2` 分别按普通成员、管理员、群主呈现；自行撤回显示“尝试撤回此信息/此图片”，管理员或群主撤回他人消息显示“尝试撤回 发送者 的信息/图片”。渲染层在消息仍可见时保存图片节点快照，`memoryOnly` 恢复时按消息 ID 回填；提示文案未变化时不写 `textContent`，避免触发 `MutationObserver` 循环。管理窗口启用 `contextIsolation`、`sandbox` 并禁用 `nodeIntegration`。
